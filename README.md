# agent-vault-template

A starter Obsidian vault built to serve as **durable memory for code agents** (Claude Code, Codex). The agent reads the vault at session start to orient itself, and writes back at session end — turning stateless agent sessions into a persistent, searchable project tracker.

Includes:

- A folder layout for tasks, daily logs, per-project session notes, and a literature KB
- Auto-generated dashboard (`VAULT_MAP.md`) built from `Work/context.md` + git log
- Auto-generated daily briefing (`MORNING_REVIEW.md`) via a reusable morning-review prompt
- Pre-commit hook to block `.DS_Store` and sensitive ID patterns
- Claude Code hook for automated literature inbox processing
- A complete walkthrough in [`TEAM_GUIDE.md`](TEAM_GUIDE.md)

---

## Quickstart

1. **Use this template** → create your own vault repo.
2. Clone it locally and open the folder as a vault in Obsidian.
3. Run the installer:
   ```bash
   ./scripts/install.sh
   ```
   Activates the pre-commit hook and marks scripts executable.
4. Edit `.githooks/pre-commit` — replace `SENSITIVE_PATTERN` with regex matching the ID shapes your data uses.
5. Skim `AGENTS.md` and adjust the ownership table / conventions to your workflow.
6. Add the session protocol to your global agent config so every session reads/writes the vault automatically:
   - **Claude Code:** in `~/.claude/CLAUDE.md`, point at the vault path and describe the read-on-start / write-on-end loop. See [`TEAM_GUIDE.md §3`](TEAM_GUIDE.md#3-how-the-agent-uses-the-vault) for the snippet.
   - **Codex CLI:** put the same protocol in your global config; Codex reads `AGENTS.md` natively, so much of it is already in place.
7. Replace the placeholder block in `Work/context.md` with a real project, copy `Work/ClaudeCode/_template/` to `Work/ClaudeCode/<your-project>/`, and you're off.
8. (Optional) Schedule the morning-review prompt (`Work/ClaudeCode/morning-review/vault-review-prompt.md`) via cron, GitHub Actions, or run it by hand each morning. It rewrites the placeholder `MORNING_REVIEW.md` at the vault root.

## Post-setup cleanup

Once your first real project is in and you've seen the morning-review run, prune the template's example artifacts:

**Replace (don't delete the file):**
- `Work/context.md` — remove the `example-project` TODO block once you have at least one real project block.
- `Literature/INDEX.md` — replace the "Topic A / Topic B" placeholder categories with real topic headings.

**Safe to delete once you've seen them:**
- `Literature/example-paper.md` — reference template for the literature note format. Delete once you've created your first real paper note (or keep as a frozen reference; your call).
- `VAULT_MAP.md` — ships with the smoke-test output from template development. Regenerates correctly on first morning-review run, so you can also just leave it.
- `MORNING_REVIEW.md` — placeholder text gets overwritten on first morning-review run. No action needed.
- `README.md` (this file) and `TEAM_GUIDE.md` — only useful for adopters; once your vault is set up you can delete them. Many people keep `TEAM_GUIDE.md` around as onboarding for teammates or future-you.

**Don't delete:**
- `AGENTS.md`, `LICENSE`, `.githooks/`, `.claude/`, `scripts/`, `Work/ClaudeCode/_template/`, `Work/ClaudeCode/morning-review/vault-review-prompt.md` — load-bearing.

**Optional removals — if you don't use that pattern:**
- `Literature/` (entire tree + `.claude/hooks/process-literature-inbox.sh` + the hook entry in `.claude/settings.json`) — if you don't track papers.
- `Work/ClaudeCode/morning-review/` — if you won't schedule a morning review.
- `docs/`, `Notes/` — if you don't use those folders.
